## ① 오늘의 운세 크래시 (`Unexpected end of JSON input`)

**원인**: `fortune/result` API가 내부 예외로 **500·빈 응답**을 반환했는데,
프론트가 그 빈 응답에 `r.json()`을 호출 → JSON 파싱 실패.

**진짜 근본 원인은 둘 중 하나일 가능성이 높습니다**:
- **DB에 별자리/타로 데이터가 안 들어있음**(seed 스크립트 미실행), 또는
- **`ANTHROPIC_API_KEY`가 없거나 틀림** → Anthropic 호출이 throw

**이번 수정**: API를 `try/catch`로 전체 감싸 **무슨 일이 있어도 JSON 반환**,
프론트도 `res.text()` 후 파싱 + 에러 화면 폴백. 이제 크래시 대신
"결과 취득 실패" 화면이 뜨고, **터미널에 진짜 원인이 로그로 찍힙니다**
(`[fortune/result] fatal: ...`). 그 로그를 보면 DB인지 키인지 바로 판별됩니다.

> ✅ 확인: 적용 후에도 결과가 안 나오면, 터미널의 `[fortune/result] fatal:` 로그를 확인하세요.
>   - `ANTHROPIC_API_KEY` 관련이면 → `.env.local`에 키 확인
>   - Supabase/DB 관련이면 → seed 스크립트로 데이터 적재 확인

## ④ DB 캐싱 대안 (비용 절감) — 상세

### 핵심 아이디어
> 같은/비슷한 입력은 **이미 만든 AI 문장을 재사용** → Claude 호출을 줄인다.

### 어디에 적용되나 (중요)
현재 AI를 호출하는 곳은 **딱 두 군데**뿐입니다:
- 오늘의 운세: **종합문(summary) 2~3문장** 1회
- 한다·안한다: **조언(advice) 한마디** 1회

별자리·타로·혈액형 텍스트는 이미 전부 DB에 있어서 **AI를 안 씁니다**.
즉 캐싱 대상은 이 두 AI 문장뿐이라, 절감 효과와 구현 범위가 명확합니다.

### 방식: 입력을 정규화한 "캐시키"로 저장/조회
`03_migration_ai_cache.sql`로 `fortune_cache` / `decision_cache` 테이블 생성.
`route-WITH-CACHE.ts`처럼 **① 캐시 조회 → HIT면 재사용(호출 0) → MISS면 생성 후 저장**.

**오늘의 운세 캐시키** = `날짜 + 별자리 + 혈액형 + 성별 + 타로3장(정/역)` 해시
- 날짜를 넣어 하루 단위로 갱신(운세는 "그날"의 개념이 자연스러움)

**한다·안한다 캐시키** = `판정 + 카드 + 정규화된 질문` 해시

### ⚠️ 냉정한 현실 — "비슷한 내용" 매칭의 함정
당신이 말한 "비슷한 내용이면 재사용"은 두 단계로 나뉩니다:

1. **완전 동일 입력 재사용** (쉬움, 지금 SQL로 가능)
   - 타로 3장까지 똑같을 확률은 낮아서 **오늘의 운세는 캐시 적중률이 낮습니다**.
   - 반면 **한다·안한다는 원카드 + 판정(3종)**이라 적중률이 훨씬 높습니다.
     같은 카드·같은 판정이면 질문이 비슷할 때 재사용 가능 → **여기가 절감 효과 큼**.

2. **"비슷한" 질문 매칭** (어려움)
   - "고백할까?" vs "고백해도 될까?"를 같다고 보려면 **의미 유사도(임베딩)**가 필요.
   - 임베딩도 API 호출이라, 배보다 배꼽이 클 수 있음. **초기엔 비권장**.
   - 대신 **질문 정규화**(공백·문장부호 제거, 소문자화)로 "거의 같은 질문"만 묶는 게 현실적.

### 권장 적용 순서 (내 판단)
1. **지금**: 캐싱 없이 크래시부터 잡고 서비스가 돌게 함(이번 fix2의 ①~③).
2. **트래픽이 붙으면**: `decision_cache`부터 적용(적중률 높음, 구현 쉬움).
3. **비용이 실제로 문제되면**: `fortune_cache` + 질문 정규화까지.
4. **임베딩 유사도**는 규모가 커진 뒤에만 검토(그 전엔 과잉 설계).

> 💡 사실 더 큰 절감은 **프롬프트 캐싱**(이미 적용됨)과 **max_tokens 축소**입니다.
>   AI 호출당 200토큰 수준이라, 하루 수천 건 전까진 비용이 크지 않습니다.
>   캐싱은 "규모가 커졌을 때의 대비책"으로 준비만 해두는 걸 권합니다.
